# Interview Playbook

## 1. 项目标题

多 Agent 求职研究助手

## 2. 简历描述

基于 Python FastAPI 和 React 改造多 Agent Demo，构建面向求职场景的智能研究助手。系统通过 Query、Insight、Report 三个轻量 Agent 顺序协作，完成岗位与公司扫描、候选人视角分析和结构化求职报告生成；前端基于 SSE 实时展示 Agent 状态，后端支持基于 `session_id` 的 best-effort 多轮会话记忆。

## 3. 三分钟介绍稿

这个项目最初是一个多 Agent 分析 Demo，我把它收缩并重新包装成了一个面向求职场景的研究助手。用户输入岗位、公司或者面试准备目标后，请求会先进入 FastAPI，再交给 `ForumEngine` 统一编排 3 个轻量 Agent 顺序执行。

第一个 Agent 负责岗位和公司信息扫描，提取目标画像和准备重点；第二个 Agent 从候选人视角总结岗位要求、项目表达和高概率追问；最后由 ReportAgent 把前面的结果整合成结构化求职研究报告，输出岗位分析、面试问题预测和行动清单。

工程上我重点补了两个点。第一个是流式状态链路，后端通过 SSE 按 `meta/status/message/done/error` 发送事件，前端按事件更新 Agent 执行状态，所以演示时可以看到完整的编排过程。第二个是多轮记忆链路，系统会基于 `session_id` 读取最近会话消息，并把历史片段拼接到当前 query 中，同时保证数据库不可用时主流程仍然能降级运行。

## 4. 高频追问

### 为什么拆成多个 Agent，而不是一个大 Prompt

因为这个项目更强调职责分工和可观测性。岗位扫描、候选人洞察和最终报告生成的目标不同，拆开后每一步的输入输出更清晰，也更容易调试和展示。

### 为什么采用顺序执行，而不是并行

v1 重点是稳定跑通和让数据流清晰。ReportAgent 依赖前面几个 Agent 的结果，顺序执行更容易保证上下文一致，也更便于前端按步骤展示。

### 会话记忆怎么实现

系统为每次会话生成 `session_id`。后续请求携带这个 id 时，后端会从会话存储读取最近若干条消息，整理成历史片段并拼接到当前 query，形成 best-effort memory。

### SSE 为什么适合这个场景

因为这里是单向流式展示，后端只需要持续把状态和结果推给前端，不需要复杂的双向通信。相比轮询，SSE 更轻量，也更适合逐步展示多 Agent 编排过程。

### 这个项目最大的工程问题是什么

为了快速跑通，我把主链路收缩成轻量 Agent 并减少了外部依赖。正式工程里仍然需要把规则型输出升级成更稳定的模型调用，并补上更清晰的依赖隔离、超时控制和自动化测试。

## 5. 你要掌握的知识框架

- API 层
  - 请求如何进入 FastAPI
  - REST 和 SSE 的差别
- 编排层
  - `ForumEngine` 如何调度多个 Agent
  - 顺序编排为什么适合 v1
- Agent 层
  - 每个 Agent 的职责边界
  - 上游输出如何被下游消费
- 状态层
  - 前端如何消费 `status/message` 事件
  - 为什么这是事件驱动状态更新
- 记忆层
  - `session_id` 如何续用
  - 为什么会话存储要做 best-effort
